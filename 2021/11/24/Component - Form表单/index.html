Component - Form表单

Posted on 2021-11-24 Edited on 2021-12-16

在众多组件实现中，Form 表单组件可能算是比较困难的一个，因为他需要包含对输入内容的管理，校验，重置以及提交等功能。在实现的过程中，收获了许多。

组件形态

为了更好表达 H5 的语义化，同时模仿 ant design 的实现，可以有以下初步的设计：每一个表单子项使用 <FormItem> 标签包裹，里面可以选择不同的输入项如 <Input> ，<CheckBox> 等

API

Form

对于 Form 组件来说，他负责的主要是收集各个子项的 value ，然后在 提交 或者 重置 的时候做批量的操作。

方法 作用
onFinish 表单提交回调函数
onFinishFail 表单提交失败回调函数
initialValue 表单子项默认值

FormItem

FormItem 组件主要维护一个表单子项的内容，主要有以下 props

方法 作用
name 表单提交时的key
label 子项显示在左侧的label值
required 是否为必需项
rules 校验规则，包括规则以及提示信息
triggerChange 引发子项内容改变的事件
triggerValidate 引发子项校验的事件

状态管理

因为 Form 组件需要管理众多的 FormItem 组件，所以状态管理这一块很容易就想到用 context 解决。但是以前往 context 里注入内容的时候，都可能是直接把众多变量和方法写在父组件里。

现在做进一步的抽象，把状态管理中的各种数据和操作委托给一个类 FormStore 处理。然后把需要暴露的方法先保存在一个 formInstanceApi 数组中，再通过 Function.bind() 把类中的方法转移到特定对象中。

据此，这样就可以完成一个 useForm 的 hooks ，在需要 FormStore 对象的组件里调用，并注入到相应 context 中。

总的步骤如下：

  1. 所有 api 方法名保存到 formInstanceApi 数组下
  2. 将 FormStore 中对应的方法通过 bind() 方法转移
  3. 完成 useForm ，使用 useRef 将 FormStore 实例缓存
  4. 创建 Form 的 context 上下文
  5. 使用 useForm 获取 FormStore 的实例 formInstance
  6. 将 formInstance 注入到 FormContext 中

组件编写

有了以上的设计，就可以开始编程了，我这里选择先编写组件，而非完成 FormStore，因为我想在构建组件的过程中，遇到困难的时候，再让 FormStore 帮助我实现，这样一来可以往这个类里慢慢添加方法，而非在一开始就费尽心思地去思索到底需要包含哪一些方法。

Form

FormStore职责

按照上面的 API 构思，已经非常清楚 Form 组件的 props 里面包含什么了。

  1. onFinish 和 onFinishFailed 都是在提交的时候执行的回调函数。
    所以对应 FormStore 也应该有一个 submit 方法，在这个方法里，会执行 onFinish 回调函数。
    另外，需要对 Form 的所有子项进行校验，如果当中有一项出错，那么就需要相应地执行 onFinishFailed。

  2. initialValue 是子项的默认值，这个也应该委托 FormStore 来保存。

  3. 另外 Form 组件还提供了 重置 功能，所以 FormStore 还应该包含一个叫 resetFields 的方法，在这个方法里对所有子项的值清空。

方法/变量 作用
setCallback 保存onFinish和onFinishFailed
submit 执行子项校验，并按需执行回调函数
defaultFormValue 子项默认值
resetFields 重置所有子项的值

Form.js

知道了 FormStore 需要解决什么问题后，就可以编写组件了。除了子组件使用 context 包裹之外，这里还使用了 useImperativeHandle 将 ref 进行转发和暴露内部方法，方便开发者直接操作 ref。

FormItem

FormStore职责

按照上面 FormItem 的 props 信息，需要把每一个子项的信息（包括value，rules…）整理成一个对象，保存在 FormStore 的 model 中。

然后每个 FormItem 在初始化的时候，都要往 FormStore 里 register ，在卸载的时候 unregister 。还有当组件内容改变时，需要实时更新或者校验对应的 value 值。所以 FormStore 有以下职责：

方法/变量 作用
registerValidateFields FormItem进行注册
unRegisterValidate FormItem进行卸载，清空对应信息
setFieldsValue 设置FormItem的value值
validateFieldValue 校验该FormItem
getFieldValue 获取该FormItem的value值
getFieldModel 获取该FormItem的model值

触发强制更新

在这里可以发现一个问题，上述的 api 中只谈到了更新 FormStore 的操作，可是却从来没有更新过视图。毕竟数据流是单向流动的，为了解决这个问题，可以先从 useState 这个 hook 中获取一个触发组件更新的函数，然后待 FormStore 更新完毕后，再触发该函数。

而该函数因为要保存起来，所以可以使用 useMemo 将其缓存，然后在组件挂载并注册的时候，将其传入，保存在 FormStore 的 control 变量中

可控组件

因为 FormItem 包裹的子组件已经是涉及内容输入的组件了，我们希望直接在 FormItem 这一层上进行内容管理，并触发 FormStore 的更新，因此使用 React.cloneElement 将 children 变为可控组件。

内容主要为：

  1. 为 children 绑定 triggerChange 的事件监听函数，并派发 value 更新信息。
  2. 为 children 绑定 triggerValidate 的事件监听函数，派发校验信息，如果和 triggerChange 是同一个事件，还需要同时执行前面绑定的监听函数，防止函数被覆盖。
  3. 从 FormStore 中获取对应的 value 值，作为 props 传给子组件。

FormStore

完成了以上组件，现在终于来到 FormStore 的实现了，有了以上编写组件过程中的需求，该类其实已经有了基本的雏形，现在再来总结一下：

成员变量与方法

变量 作用
model 对应每个表单子项的数据
control 对应每个表单子项的强制更新函数
callback onFinish与onFinishFailed
defaultFormValue 表单子项默认值
方法 作用
getForm 返回一个对象，包含所有暴露的 api 方法
setCallback 保存onFinish与onFinishFailed
registerValidateFields 表单子项注册函数
unRegisterValidate 表单子项卸载函数
resetFields 重置表单所有 value 值
setFieldsValue 设置表单某个子项的 value 值
getFieldModel 获取表单某个子项的 model 值
getFieldValue 获取表单某个子项的 value 值
submit 表单提交函数

以上表格里已经包含了 FormStore 的大部分成员变量及方法。唯独缺少的是 更新派发 与 校验 相关的函数，下面就对这两部分展开来讲。

dispatch

如果仔细观察上面的 FormItem.js 代码，会发现在使用 FormStore 的时候，都是通过一个 dispatch 方法，而非直接调用 getFieldValue 这些函数。

这一点是模仿了 redux 的思想，对数据的操作都通过派发一个 action 来操作。

校验

对于表单每个子项的校验结果，都分为了三个状态，这一点和 Promise 是一样的，分别是 resolve ，reject 和 pendding。

每当重新设置了 value 后，都会把状态重新置为 pendding ，并触发对应的组件强制更新函数。

penddingValidateQueue

而校验函数 validateFieldValue 校验完成后不会每一次都触发强制更新，因为对于 校验状态 没有改变的情况来说是浪费性能的。

另外为了应对批量校验的情况，有可能会触发多次组件渲染，性能损耗严重，再者校验的视图更新不是那么重要。所以建立了一个队列，每次在校验完成后，不立刻强制更新，而是把对应的强制更新函数 push 到队列，然后开启 schedule 调度。使用 unstable_batchedUpdates 作批量更新。

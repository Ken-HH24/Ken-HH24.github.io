IntersectionObserver实践

Posted on 2021-11-29 Edited on 2021-12-09

Preview

秋招面试的时候有一道常考的题目，那就是图片懒加载如何实现，尽管如今部分浏览器已经支持了原生的 loading 懒加载属性，大部分人的回答依然是使用 scrollTop ，offsetTop 等几个元素高度属性来解决问题。归根结底，就是需要检测元素是否出现在用户可视范围之内，而对于这个功能，浏览器也已经有 api 支持了。

IntersectionObserver

这个 api 就是 IntersectionObserver 。它能够帮助我们监听目标元素，是否与另外一个元素相交，然后执行对应的回调函数。该类的构造函数形式如下：

observer 为调用构造函数时返回的值
IntersectionObserverEntry 成员的定义为：

ImageLoader

有了上面可以动态监听元素是否可见的 hook ，可以利用它来打造一个具有图片懒加载功能的 Component 。

  1. 把组件状态分为 visible 和 loaded 两种状态。
  2. 利用 useIntersectionObserver 对 <img> 标签作可见监听。
  3. 在用户可见时执行回调函数，设置 visible 为 true ，同时对图片 url 发送请求。

useActive

之前阅读过一篇有关阿里 BI 平台的文章，前端要呈现各种图表，布局方案有多种，在组件初始加载，数据未获取的情况下选择了全量渲染方案。
但是由于用户的每一次选择都可能导致呈现的图表不一样，图表组件加载百万级的数据稀松平常，这个时候就需要利用 判断DOM是否在画布中 这个技术来执行 按需渲染

按需渲染 的实现最终还是归为 IntersectionObserver ，与前面的 hook 不同，这次的 useActive 会返回一个 active 值，然后将该值交由组件处理并决定是否重新渲染。其实差别就在于回调函数的处理上：

  1. hook 内部生成回调函数，根据元素是否可见决定 active 的值。
  2. 当 active = false ，外部组件会抑制渲染，但是此时数据可能会发生变化，当切换为 true 时，可能会造成数据视图不一致的结果。因此需要利用一个 forceUpdate 在切换为 true 的时候进行强制更新。

总结

本次主要是利用 IntersectionObserver 这个 API 实现了两个 hook ，以及一个图片懒加载组件以及一个按需渲染组件，有以下收获：

  1. 掌握了 IntersectionObserver 的基本使用。
  2. 通过 onLoad 事件解决图片缓慢加载时的割裂感。
  3. 通过 HOC 和 Class Component 的方式来限制组件的渲染。

React中的渲染优化

Posted on 2021-11-20 Edited on 2021-11-21

Suspense

作用：Suspense 提供了一种方式让组件的数据请求完成后，再渲染该组件

在这里可能会产生疑问，如果仅仅是针对数据请求的优化，那么 “我在全局里加一个 isLoading 变量不就好了吗？”。但是二者还是有一定差别的。

  1. 首先，添加 isLoading 变量在数据请求前后组件都渲染了一次，无法达到：组件挂起 -> 数据请求 -> 组件渲染这个过程。
  2. 另外在 React 官方文档中详细介绍了 Suspense 组件的存在意义。下面代码中，在 fetchUser 没有 resolve 之前，fetchPosts 是不会执行的，在一定程度上失去了异步请求的意义，官方称这种情况为 waterfall

而官方还列举了另外一种方式，便是采用 Promise.all ，但是这意味着只有当所有请求完成后，用户才能看到最后的页面。如果应用日渐庞大，数据量急剧上升，那么这个 Promise.all 将变得臃肿

而 Suspense 组件会将正在进行数据请求的子组件置为 suspense 状态。原理是利用try catch ，进行数据请求时会把该请求的 Promise 捕捉，等待 resolve 后再进行渲染，可以利用 componentDidCatch 这个钩子函数实现类似的 Suspense 功能

Suspense + React.lazy

React.lazy 接受一个函数，这个函数需要动态调用 import() ，必须返回一个 Promise ，该 Promise 需要 resolve 一个 default export 的 React 组件

其内部原理其实就是维护一个 result ，当 payload 对应的 Promise 还没有 resolve 的时候，会抛出一个错误，这个时候可以被 Suspense 组件捕捉到，然后等待 resolve 后再把组件挂载到 default 属性上

利用二者可以达到将数据请求和组件分离的 code split 效果，并实现 异步组件

Virtual List

虚拟列表 是一个常见的前端优化技术，当渲染列表节点过多，比如超过一万个的时候，通常就需要用 虚拟列表 来进行优化，防止页面出现卡顿。

虚拟列表 的核心思想是监听用户当前可视区域范围，选择性地渲染节点。下面使用 React 来进行组件的封装

HTML部分

组件的 HTML 部分主要分为 scroll_box - 滚动区域 ，scroll_hold - 撑开容器的元素 和 scroll_content - 真实渲染的节点。

组件挂载

而组件的变量有 dataList - 节点数据， position - 当前渲染列表的起始与结束位置， height - 容器高度 和 itemHeight - 节点高度。当然还有一系列 ref 变量

当组件挂载的时候，先通过容器的高度，以及节点高度，计算可以渲染的节点数量 renderCount 。容器高度可以通过 ref 的 offsetHeight 获取，随后更新一下 scrollInfo

知道以上的信息之后，便可以模拟出 dataList ，以及更新 position 变量了。当然，如果该组件投入到应用中，dataList 可以作为 props 变量传入。

上述的两个过程都写在 useEffect 里面

监听滚动

然后便是编写 handleScroll 函数来监听滚动动作了。通过 scrollTop 获取当前滚动条的位置，开始位置便是 scrollTop / itemHeight ，结束位置则是加上 renderCount ，如果 start 和 end 中有一个与当前 position 不相同，那么执行更新。

另外，为了表现出滚动动画，改变可视区域的 translate3d 属性

执行渲染

最后，在 scroll_content 里渲染 dataList.slice(start, end) 的数据。而 scroll_hold 因为要撑开容器，所以高度设置为 dataList.length * itemHeight ，而 scroll_box 则设置为预设的容器高度。

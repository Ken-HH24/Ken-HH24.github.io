源码实践 - axios

Posted on 2021-12-10

preview

axios 是前端常用的网络请求库，在 github 上有将近 90k stars 。翻开源码，其核心实现部分也并不复杂，个人觉得精髓在于 interceptor 拦截器 和 Adapter 适配器的设计。

axios

Interceptor

完成了上面的工作之后就可以通过 axios 进行正常的网络请求了，但是有时候需要在请求之前对数据作某种处理，或者在获得数据后对数据进行过滤。这些需求就衍生出了 Interceptor 。

InterceptorManager

源码里的 interceptor 都交给 InterceptorManager 管理，而其内部结构也很简单，只有一个 handlers 数组保留所有 interceptor ，然后 Axios 里对于请求过程和获取过程对应地建立两个 InterceptorManager

而当用户需要使用某个 interceptor 的时候，会调用 InterceptorManager.prototype.use 方法，传入两个参数，分别是请求成功和失败的 interceptor 函数。

在 use 函数内会将两个函数包装成对象放在 handlers 数组里，然后返回该 interceptor 的下标。

但是 use 方法和之前的 get 方法有一个共同的问题，那就是 use 函数处在 InterceptorManager.prototype 上，而 axios 是一个 Axios.prototype.request 对象，不能直接调用 use() 方法。

pipline

上面只是提到了 interceptor 如何注入到一个 axios 请求里，那么当真正执行 request 函数的时候，axios 是如何把众多拦截器串联在一起的呢？

axios 源码把过程分为了向服务端发送请求和从服务端获取数据两个过程，建立一个 chain 数组，初始值是一个 dispatchRequest 函数和 undefined ：

然后把 interceptors.request 的每一个对象，都推到 chain 数组的开头，把 interceptors.response 的每一个对象推到末尾，在该过程中还把 fufilled 和 rejected 两种状态分开了

最后生成一个 resolve(config) 的 Promise ，进入循环，不断从 chain 数组中取出每个 interceptor 的 fullfilled 和 rejected ，调用 then 方法获取最新的 Promise ，进而返回出去：

在上述过程中，所有 interceptor 被组装起来，像管道 pipline 一样对请求进行处理，共分为了三个阶段：

  1. 向服务端请求过程：此时 Promise 里能够访问的一直是 config 变量，在每个函数处理完成后，还要把新的 config 返回出去
  2. 调用 dispatchRequest ：此时拿到最后生成的 config ，基于 XMLHttpRequest ，进行真正地请求
  3. 从服务端获取数据：此时拿到了刚刚 dispatchRequest 返回的 Promise ，而里面 resolve 的是 data ，所以在后面调用 then() 的时候，Promise 里能够访问的是 data 变量

Adapter

进一步研究 dispatchRequest 这个函数，会发现他里面真正调用的是 adapter 这个函数，这就是 axios 的另一个特点，为了应对浏览器端与服务端的差异，采用了 适配器模式 来对网络请求进行不同的处理。在浏览器内使用的是 XMLHttpRequest ，而在服务端使用的是 Node 的 http 。另外，使用者也可以根据自己的需求，来创建适配器 adapter。

axios 源码里还有一个颇为优秀的点就是其代码复用能力，前面把 get 等方法放在 Axios.prototype ，和把 Axios 对象的属性放在 axios 上时，都有遍历某个对象的过程，于是源码里把这一过程抽象出来，放在了 utils.forEach 里，有点 迭代器模式 的味道，也和 semi design 里的 traverseDataNodes 函数实现风格相近：

总结

通过阅读 axios 的源码，主要有以下收获：

  1. interceptor 拦截器是如何实现的
  2. 面对不同平台的网络请求， axios 采用了 适配器模式 创建 adapter 解决
  3. 在浏览器端的网络请求里，底层依赖的依然是 XMLHttpRequest
  4. 最后的网络请求都走向 request 函数，但是通过将 get 等方法转到 prototype ，prototype 转到 axios (实际上也是request函数) ，提供了不同的 api 请求方式
  5. utils.js 里把遍历过程抽象出来，实现更强大函数的方法

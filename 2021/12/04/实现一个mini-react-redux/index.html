实现一个mini-react-redux

Posted on 2021-12-04 Edited on 2021-12-05

preview

上一篇完成了阅读 redux 源码并动手实现了一个 mini-redux 。redux 是一个面对 JS 的状态管理库，当投入到 React 里使用中时，如果仅从 subscribe 函数上进行展开使用，会变得十分麻烦。

react-redux 的出现使得在 React 能够更好地使用 redux ，通过 Provider 组件将整个 store 注入，然后当子组件需要使用到 store 里的数据时，通过 connect 函数，编写 mapStateToProps 和 mapDispatchToProps ，就可以将目标 store 数据注入到目标组件的 props 中。

Question

在网上经常可以看到，React 中的 useContext 和 useReducer 已经可以基本满足状态管理的需求了，那么为什么还需要 react-redux ，直接使用 useContext 并注入 redux 中的 store 不就好了吗？

首先使用 useContext 就表明不可直接将数据注入 props 中，当然使用 hooks 也有他的好处。但是由于 redux 的状态 state 是通过闭包保存在 store 里面的，所以当调用 dispatch 改变 state 时，React 的 Context API 是检测不到 store 的变化的，也无法进行更新，因此必须实现一套新的 API 供 React 使用 redux 。

Provider

首先来实现 Provider 组件，他的核心与 React 中的 Context 类似，就是注入目标数据 store ，供子组件使用。

翻开源码会发现，Provider 组件依然使用了 React Context ，主要做了以下内容：

  1. 全局通过 createContext 创建一个 ReactReduxContext 对象
  2. 在 Provider 函数组件里创建一个 subscription 对象，方便后续状态更新的派发
  3. 组件最后以 Context.Provider 渲染，注入 store 和 subscription

Subscription

createSubscription

前面提到 React useContext 的问题是无法监听 store 的状态改变，而 subscription 就是解决该问题的，底层的原理就是利用 redux store 暴露出来的 subscribe 函数。

subscripttion 的职责如下：

  1. 使用闭包维护一个 Listeners 链表结构，在状态更新的时候依次执行链表中的回调函数
  2. 订阅，并使用闭包保存解除订阅函数
  3. 添加新的回调函数

特别需要注意的一点就是 createSubscription 函数还可以传入一个父级 fathersubscription 对象，当父级 fathersubscription 存在时，subscription 会将自己的事件处理函数往 fathersubscription 里订阅，而不存在时则会调用底层的 store.subscribe() 达到监听 store 状态更新的目的。 fathersubscription 存在的意义是应对 Provider 里使用 connect 的组件多级包裹的情况，所以需要借助父子级关系构建树状结构。

createListenerCollection

Listeners 是一个链表数据结构，其功能为：

  1. 维护链表的头尾指针 first 和 last
  2. 注册一个新的 listener ，即在链表末尾添加节点，同时返回解除监听函数
  3. 具有清空链表功能

onStateChange

在这里看到 subscription 的 handleChangeWrapper 函数可能会有点懵，因为里面调用了 onStateChange ：一个还没有定义的函数。

在我看来，handleChangeWrapper 和 onStateChange 的意义是：在创建该 subscription 的时候并不知道对于状态更新需要采取什么策略，所以没有把 onStateChange 写死，而是交由外部进行动态赋值。

而在 Provider.js 里，对于 subscription 的处理是这样的：

  1. 调用 createSubscription 没有传入第二个参数，说明是调用 store.subscribe ，该 subscription 监听了 store 的状态改变
  2. onStateChange 赋值为 notifyNestedSubs ，意味着 store 状态更新时，会执行 listeners.notify() ，把所有回调函数执行一遍
  3. 执行 trySubscribe 会构建一个全新的链表，以后使用 store 的组件的更新函数就可以放在这个链表里，结合 onStateChange 完成视图更新

小结

至此，Provider.js 和 subscription.js 已经完成，其设计思路如下：

  1. Provider 组件通过全局变量 ReactReduxContext ，采取 Provider 模式向子组件注入 store 和 subscription
  2. 在 Provider 组件内创建一个 subscription 对象，通过调用 store.subscribe() 监听 store 的变化
  3. 该 subscription 对象内部会维护一个链表数据结构，里面可以放入回调函数，如组件更新函数，以供视图更新
  4. 当 store 发生改变时，会依次执行链表内的回调函数，达到状态改变，视图更新的目的

connect

connect 函数的职责是利用传入的 mapStateToProps 和 mapDispatchToProps 函数将 store 里面的数据映射到目标组件的 props 中，源码里函数的实现过程较为复杂，下面分步骤来讲：

mapToProps 处理

首先是对于参数的处理，由于用户传入的 mapToProps 未必就是函数，所以需要分情况讨论。这里源码采用了类似 职责链模式 的设计模式，将各种情况的处理函数放在一个数组中，然后依次把 mapToProps 放入进行检测。

不过本文因为实现的是 mini 版本，该过程就不在此实现了，拿第一个参数的处理过程举例：

match

首先把各个处理函数放入 defaultMapStateToPropsFactories 数组中，然后把 mapStateToProps 传入 match 函数，进行遍历调用，直到得到结果为止

wrapMapToPropsFunc

当检测到 mapStateToProps 是一个函数时，会再把它交由一个 wrapMapToPropsFunc 处理。在这里面会返回一个新的函数，使用 代理模式 将 proxy 取代原来的 mapStateToProps 。

proxy 会添加一个 dependsOnOwnProps 属性表示 mapStateTo 是否依赖于目标组件自己的 props ，该值会通过 mapStateToProps 函数的参数长度决定。

源码里 proxy 的 mapToProps 指向经常变化，需要注意一下。

HOC

处理完 mapToProps 后，connect 函数会返回一个新的函数，该函数接收目标组件为参数，然后对其进行 HOC 处理，把 store 里的数据注入到 props 中，返回新的组件。

props处理

源码里考虑到有用户自定义 Context 的存在，所以先通过 props.context 将其提取出来。当然源码里还考虑到了 ref 转发的问题

ContextToUse

在前面获取了 props.context 后，就可以通过其是否存在决定应该使用哪一个 context 了，如果不存在，则使用全局的 ReactReduxContext

useContext

获取到真正 context 后，通过 React 里的 useContext 获取到 value 值，并把 store 提取，另外建立一个变量 didComeFromProps 标记本次使用的 context 是否来自 props

childPropsSelector

childPropsSelector 的作用是合成组件的最终 props ，在源码里，因为考虑到渲染模式，配置选项等的不同，所以采用 工厂模式 利用 finalPropsSelectorFactory 来构建该函数。

在 finalPropsSelectorFactory 里，会调用 mapToProps 等方法生成真正的 proxy ，随后利用闭包存储各个 state 和 dispatch 变量，最终生成的 childPropsSelector 里会根据 调用次数 ，state 或者是 props 改变，采取不同的策略合成最终 props

而在实现的 mini version 里就不采取如此复杂的逻辑了，直接调用 mapToProps 并合成：

创建subscription

subscription 的作用就是在 store 状态更新时通知视图更新的，所以每一个 HOC 组件里都要注册一个新的 subscription 。

而因为是子组件，所以不再调用 store.subscribe 进行监听了，前面提到的 createSubscription 的第二个参数起到了作用。在这里把现在注册的 subscription 添加到 contextValue.subscription 的链表里去。说明当前组件的视图更新依赖于 上下文环境 中的 subscription

生成新的上下文环境

前面提到当前组件的视图更新依赖于他处于的 上下文环境 中的 subscription ，而该组件的子组件视图更新就依赖于当前组件的 subscription ，所以需要生成新的 context

生成最终合成props

有了一系列缓存变量以及前面用 工厂模式 生成的 childPropsSelector 函数，就可以生成最终的 合成props 了：

checkForUpdates

前面创建了 subscription ，但是 subscription 的更新依赖于 onStateChange 函数，所以还需要进行 subscribe 。

源码把这一段逻辑放在了 useEffect 里面，定制一个 checkForUpdates 函数，主要会干以下事情：

  1. 利用闭包创建变量 didUnsubscribe 防止重复订阅
  2. 获取 store 里的最新状态 latestStoreState
  3. 合成最新的 newChildProps
  4. 如果最新的 childProps 与上一次的缓存一样，那么本组件不更新，通过 notifyNestedSubs 通知子组件进行视图更新
  5. 否则更新一系列缓存变量，然后通过 forceUpdate 进行视图更新，其中 forceUpdate 为 useReducer 返回的 dispatch 函数

把 checkForUpdates 保存在 onStateChange 后，执行 trySubscribe 进行注册，并保留 unSubscribe 函数

captureRestProps

但是上面的 checkForUpdate 函数中存在一些问题，在本组件进行视图更新的时候，也应该相应地通知子组件更新，源码把这一段逻辑放在了 captureRestProps 里，也使用 useEffect 包裹，当检测到 childPropsFromStoreUpdate 有缓存变量时，通过 notifyNestedSubs 通知子组件进行视图更新

render

有了新的合成 childProps 和 overriddenContextValue ，HOC 可以返回最后的渲染组件了，将 actualChildProps 注入到 WrappedComponent ，overriddenContextValue 注入到 ContextToUse.Provider 里。

React.memo

在 connect 函数的最后，还应用了 React.memo 将前面的 HOC 组件包裹起来，这样做的目的是因为在 React 里父组件的更新往往会驱使子组件更新，需要让各个 HOC 的视图更新独立。

而在源码中还通过 hoistStatics 把静态方法挂载到 HOC 上，还有关于 ref 的转发处理。

总结

至此，一个 mini-react-redux 就已经完成了，在实现的过程中，主要有以下收获：

  1. 明白了 React useContext 不能满足 redux 使用要求的原因
  2. 使用 更新订阅模式 实现对 redux 状态更新，然后视图更新的需求
  3. 使用 职责链模式 对函数参数进行分类讨论
  4. 使用 工厂模式 以应对不同的配置选项，生成 mapToProps 不同的情况
  5. 使用 代理模式 对 mapToProps 处理，添加 dependOnOwnProps 属性达到性能优化
  6. 使用 链表 数据结构维护视图更新的回调函数
  7. 大量使用 useRef 和 useMemo 来缓存变量，进行性能优化，避免不必要的重复渲染

Component - DraggableTree可拖拽树形控件

Posted on 2021-12-13 Edited on 2021-12-16

preview

在上一次组件实践中模仿了 semi design 完成了 Tree树形控件 的基本功能。在这基础上可以添加许多额外功能，比如 select ，checkbox 的支持。但是这些都稍显简单了，基本思路就是在 TreeNode 里添加相应的事件监听，然后根据 F / A 架构在 TreeFoundation.js 里书写逻辑，添加对应的 selectedKeys 或者 checkedKeys 即可。

而拖拽功能在如今前端里应用得很广泛，semi design 在 Tree树形控件 里也集成了该功能。在我看来，实现的难点主要为：

  1. 在拖拽事件结束后，原节点顺序被打乱，如何快速得到新的节点顺序是关键
  2. 如何在拖拽的过程中提供较良好的用户体验，比如在用户拖动到文件夹上时，该结构会自动展开。又或者添加适当的提醒让用户知道现在的拖拽结果

而以上问题都在阅读 semi design 源码，并自己动手实践后得到解决。

draggable

首先要达到可拖拽的效果，就需要使用 draggable 这个 API ，首先需要显式地在标签里添加 draggable = ‘true’ 而不能只写 draggable 。

然后为每个 TreeNode 添加 onDragStart ， onDragEnter 等事件。其中在 onDragStart 事件中，需要特别地调用 e.dataTransfer.setData('text/plain', ''); ，否则在某些浏览器内不会生效。另外在拖拽 <li> 标签的过程中会发现鼠标状态一直是一个禁止的图标，非常影响用户体验，要解决这个问题则需要在其他 drag 事件中添加 e.dataTransfer.dropEffect = 'move'; ，改为移动图标。

F / A

首先依然是贯彻 semi design 的 F / A 架构，将整个事件的处理逻辑放在 TreeFoundation.js 中，然后将对应的事件处理函数放入 Context ， TreeNode 只负责调用即可。

onDragStart

在拖拽的最开始，需要做的事情主要有两步：

  1. 记录被拖拽的节点
  2. 更新 Tree 状态

主要涉及以下状态：

state function
dragging 整个 Tree 是否处于拖拽状态
dragNodeKeys 被拖拽所有节点的keys

source code - react-redux

Posted on 2021-12-04 Edited on 2021-12-16

preview

上一篇完成了阅读 redux 源码并动手实现了一个 mini-redux 。redux 是一个面对 JS 的状态管理库，当投入到 React 里使用中时，如果仅从 subscribe 函数上进行展开使用，会变得十分麻烦。

react-redux 的出现使得在 React 能够更好地使用 redux ，通过 Provider 组件将整个 store 注入，然后当子组件需要使用到 store 里的数据时，通过 connect 函数，编写 mapStateToProps 和 mapDispatchToProps ，就可以将目标 store 数据注入到目标组件的 props 中。

Question

在网上经常可以看到，React 中的 useContext 和 useReducer 已经可以基本满足状态管理的需求了，那么为什么还需要 react-redux ，直接使用 useContext 并注入 redux 中的 store 不就好了吗？

首先使用 useContext 就表明不可直接将数据注入 props 中，当然使用 hooks 也有他的好处。但是由于 redux 的状态 state 是通过闭包保存在 store 里面的，所以当调用 dispatch 改变 state 时，React 的 Context API 是检测不到 store 的变化的，也无法进行更新，因此必须实现一套新的 API 供 React 使用 redux 。

Provider

首先来实现 Provider 组件，他的核心与 React 中的 Context 类似，就是注入目标数据 store ，供子组件使用。

翻开源码会发现，Provider 组件依然使用了 React Context ，主要做了以下内容：

  1. 全局通过 createContext 创建一个 ReactReduxContext 对象
  2. 在 Provider 函数组件里创建一个 subscription 对象，方便后续状态更新的派发
  3. 组件最后以 Context.Provider 渲染，注入 store 和 subscription
